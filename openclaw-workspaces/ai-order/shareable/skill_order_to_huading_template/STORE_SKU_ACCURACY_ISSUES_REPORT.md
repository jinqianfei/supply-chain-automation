# 门店与 SKU 准确率问题专项报告

测试日期：2026-06-11  
来源报告：`ORDER_MAPPING_ACCURACY_REPORT.md`  
专项范围：只分析本轮准确率测试中暴露的两个核心问题：

1. 门店误命中：手机号精确匹配命中同货主下另一个门店。
2. SKU 未命中/误匹配：名称模糊匹配被防误匹配规则挡住，或同名/近名 SKU 选择错误。

## 1. 结论摘要

本轮测试中，“需人工确认”的结果不视为异常，这是订单映射流程的正常安全机制。

剔除“需人工确认属于正常流程”后，当前需要重点修复的是：

| 问题类型 | 数量 | 影响 |
| --- | ---: | --- |
| 门店误命中 | 1 | 门店编码错误，但货主正确 |
| SKU 未匹配 | 1 | 有正确 SKU，但当前规则未匹配出来 |
| SKU 误匹配 | 5 | 匹配到近似或同名相关 SKU，其中部分应由候选确认或别名解决 |
| 单位类型异常 | 1 | 经复核为 GT 标注问题，不是系统单位逻辑问题 |

## 2. 门店误命中问题

### 2.1 失败样本

| 字段 | 内容 |
| --- | --- |
| 输入门店 | 廖朵朵蛋糕-开封尉氏张英杰 |
| 输入手机号 | 18530139599 |
| 期望门店编码 | KH2024100600013 |
| 实际命中门店编码 | KH2025011400028 |
| 期望货主 | HZ2024091100001 |
| 实际货主 | HZ2024091100001 |
| 匹配方式 | 手机号精确匹配 |

### 2.2 根因

门店匹配逻辑中，手机号是最高优先级。只要传入手机号并在 `store_list.phone` 中查到记录，就会直接返回该手机号对应门店。

当前执行路径：

1. 有手机号，优先执行 `_find_by_phone(phone)`。
2. 手机号命中门店后，立即返回 `match_type = phone_exact`。
3. 后续门店名称精确匹配、包含匹配、模糊匹配不再执行。

因此，这条样本没有继续用 `廖朵朵蛋糕-开封尉氏张英杰` 做门店名称校验，而是直接采用了手机号命中的另一个门店。

### 2.3 影响判断

货主命中是正确的，说明大方向没有错；但门店编码错误，会影响华鼎模板中的“门店编号”。

主流程当前只允许 `match_type == exact` 自动确认，`phone_exact` 会进入人工确认。因此从流程安全性看，这个问题不会自动放行，但会给用户展示一个偏向手机号的候选结果。

### 2.4 修复建议

手机号命中后不要直接终止，应增加二次校验：

1. 计算输入门店名与手机号命中门店名的相似度。
2. 如果相似度高，仍返回手机号命中结果，但标记为需确认。
3. 如果相似度低，继续执行门店名称匹配，把“手机号命中门店”和“名称命中门店”都加入候选列表。
4. 候选展示中标明原因：手机号命中、名称命中、地址命中、联系人命中。

建议策略：

```text
手机号命中 + 门店名相似度 >= 0.75
  -> 手机号结果作为首选候选

手机号命中 + 门店名相似度 < 0.75
  -> 不直接返回，继续跑名称匹配
  -> 合并候选，要求用户确认
```

## 3. SKU 未匹配问题

### 3.1 失败样本

| 字段 | 内容 |
| --- | --- |
| 订单商品名 | 鱼你幸福青花椒酱料（新款）—原椒麻酱料 |
| 期望 SKU | SK250214000074 |
| 数据库 SKU 名称 | 鱼你幸福青花椒酱料（新款） |
| 单位 | 箱 |
| 单位类型 | 大单位 |
| 规格 | 10包/箱 |
| 实际结果 | 未匹配 |

### 3.2 根因

SKU 匹配逻辑有做名称模糊匹配，但这条被“调料类防误匹配规则”挡住了。

当前清洗逻辑会去掉括号内容，但不会去掉破折号后的补充描述。因此：

```text
鱼你幸福青花椒酱料（新款）—原椒麻酱料
```

清洗后仍保留：

```text
鱼你幸福青花椒酱料—原椒麻酱料
```

而数据库标准名是：

```text
鱼你幸福青花椒酱料（新款）
```

对于“酱料/粉/膏”这类调料商品，当前逻辑为了防止口味误配，要求核心词更严格。例如防止：

```text
金汤酱料 -> 番茄酱料
```

这条样本里，订单名多出来的 `—原椒麻酱料` 被当成核心名称的一部分，导致核心词校验不通过，所以没有进入最终匹配。

### 3.3 影响判断

这不是“没有做模糊匹配”，而是模糊候选被防误匹配规则过滤掉。

防误匹配规则本身有价值，不能简单放宽，否则会增加调料类 SKU 错配风险。更稳妥的处理方式是补别名或增强清洗规则。

### 3.4 修复建议

优先建议加别名：

```text
鱼你幸福青花椒酱料（新款）—原椒麻酱料
  -> 鱼你幸福青花椒酱料（新款）
```

中期建议增强清洗规则：

1. 对破折号、长横线、冒号后的补充描述做识别。
2. 如果破折号前半段已能高置信匹配 SKU，后半段可作为备注/别名描述，不进入核心词校验。
3. 只对已知品牌或已知调料类 SKU 启用，避免误删真正商品名。

建议策略：

```text
原始名: A—B
如果 A 与某个 SKU 名称高相似，且 B 含“原/新款/口味/酱料”等描述词
  -> 用 A 做主匹配
  -> B 作为备注或别名证据
```

## 4. SKU 误匹配问题

本轮 SKU 错误主要分两类。

### 4.1 多候选/近名商品

| 商品 | 实际 SKU | 期望 SKU | 原因 |
| --- | --- | --- | --- |
| 免浆巴沙鱼片 | SK240803000119 | SK251101000101 | 同类鱼片近名，Layer 2 多候选并列 |
| 免浆黑鱼片 | SK240803000117 | SK250612000056 | 同类鱼片近名，Layer 2 多候选并列 |
| 免浆巴沙鱼片 | SK240803000119 | SK240803000107 | 同类鱼片近名，Layer 2 多候选并列 |

判断：

这类属于候选选择问题。底层已经识别为低置信或多候选，合理流程应交给人工确认。风险在于主入口当前没有完整透传 `confidence`、`need_confirm`、`candidates`，导致审核表可能看不到候选风险。

修复建议：

1. 主入口 `_match_sku()` 必须透传 `confidence`、`need_confirm`、`candidates`。
2. 对 Layer 2 多候选并列结果，不应作为普通已匹配展示。
3. 审核表中将多候选商品标红或标记“需选择 SKU”。

### 4.2 同名精确匹配但期望不同

| 商品 | 实际 SKU | 期望 SKU | 数据库复核 |
| --- | --- | --- | --- |
| 调面汁 | SK250327000074 | SK250327000076 | SK250327000074 是调面汁；SK250327000076 是剁椒酱 |

判断：

这条更像 GT 标注问题。数据库中：

- `SK250327000074`：调面汁
- `SK250327000076`：剁椒酱

系统匹配 `调面汁 -> 调面汁` 是合理的。

### 4.3 单位类型异常但实际为 GT 问题

| 商品 | 实际 SKU | 实际名称 | 期望 SKU | 期望名称 |
| --- | --- | --- | --- | --- |
| 深岩静腌西冷牛排 | SK250820000194 | 深岩静腌西冷牛排 | SK241225000055 | 浩然奥尔良翅中 |

判断：

这条不是单位类型算法错误，而是 GT 标注错误。

系统匹配结果：

- SKU：`SK250820000194`
- 名称：`深岩静腌西冷牛排`
- 单位：`箱`
- 单位类型：`大单位`
- 规格：`2.5kg*6包/箱`

GT 期望：

- SKU：`SK241225000055`
- 名称：`浩然奥尔良翅中`
- 单位：`件`
- 单位类型：`小单位`

商品名完全不对应，应修正 GT。

## 5. 修复优先级

### P0

1. 门店手机号命中后增加门店名称二次校验。
2. `_match_sku()` 透传 `confidence`、`need_confirm`、`candidates`。
3. 多候选 SKU 在审核表中必须明确展示为“需人工选择”。

### P1

1. 增加别名：`鱼你幸福青花椒酱料（新款）—原椒麻酱料 -> 鱼你幸福青花椒酱料（新款）`。
2. 清洗规则支持破折号后补充描述识别。
3. 修正明显错误 GT：`调面汁`、`深岩静腌西冷牛排` 相关标注。

### P2

1. 扩充门店测试集，覆盖手机号复用、手机号变更、同货主同城市多店。
2. 扩充 SKU GT，尤其是鱼片、酱料、牛排这类同名/近名/多规格商品。
3. 将“自动准确率”和“人工确认后准确率”分开统计。

## 6. 最终判断

门店错误的本质是手机号优先级过高，缺少名称二次校验。

SKU 未匹配的本质不是没有模糊匹配，而是调料类防误匹配规则过严，且商品别名/清洗规则没有覆盖破折号后的补充描述。

单位类型问题中最明显的一条是 GT 错误，不应归因于系统单位类型逻辑。
