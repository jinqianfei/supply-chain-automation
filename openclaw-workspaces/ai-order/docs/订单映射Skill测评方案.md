## 1. 背景与目的

### 1.1 背景

订单映射Skill是AI建单助手的核心能力，负责将客户订单转换为华鼎标准出库单模板。其处理流程为：

```
门店匹配 → 货主确认 → SKU映射 → 人工确认 → 生成模板
```

该Skill涉及多系统交互（数据库、规则库、模板生成器），且直接参与业务建单流程，准确率要求高。

### 1.2 测评目的

1. **验证功能完整性** - 确保门店匹配、SKU映射、模板生成等核心功能按设计正常工作
2. **建立质量基准** - 为后续迭代建立量化指标基线
3. **识别潜在风险** - 发现上线后可能出现的业务风险
4. **指导优化方向** - 通过测评数据分析，确定优化优先级

## 2. 测评范围与边界

### 2.1 测评范围

| 模块 | 测评内容 |
|------|----------|
| 门店匹配 | Layer 0精确匹配、Layer 1模糊匹配、Layer 2门店名清洗 |
| SKU映射 | Layer 0别名表、Layer 1精确匹配、Layer 2模糊匹配、Layer 3分词匹配、规格校验 |
| 规则库 | default.yaml、小江溪.yaml、王小五.yaml、创宇.yaml |
| 模板生成 | 31字段完整性、序号规则、多门店支持、格式校验 |
| 性能 | 响应时间、并发能力、资源消耗 |

### 2.2 不在测评范围内

- 第三方依赖系统（如飞书API）
- 硬件故障、网络中断等基础设施问题
- 人工建单流程

## 3. 验收标准

### 3.1 功能性验收标准

| 验收项 | 验收条件 | 验证方法 |
|--------|---------|---------|
| 门店匹配 | 输入订单后，门店100%能匹配到货主ID | 用测试集验证，输出need_confirm标记 |
| SKU映射 | 有效SKU商品映射成功率≥90% | 匹配成功数/总商品数 |
| 模板生成 | 生成的Excel符合华鼎31字段规范 | 字段完整性 + 格式校验 |
| 多门店支持 | 支持1-50个门店的订单 | 边界测试 |
| 错误识别 | 无法匹配时明确提示用户 | 人为输入错误数据验证 |
| 规格校验 | SKU规格与订单规格不匹配时，按规则处理 | 详见3.4规格校验规则 |

### 3.2 性能验收标准

| 指标 | 目标 | 测量方法 |
|------|------|---------|
| 单订单响应时间 | ≤10秒 | 计时器测量，取P95 |
| SKU查询延迟 | ≤100ms/次 | 数据库慢查询监控 |
| 网络延迟容错 | ≤500ms（含网络开销） | 端到端延迟测量 |
| 内存占用 | ≤500MB/订单 | 进程监控 |

### 3.3 业务验收标准

| 指标 | 目标 | 测量方法 |
|------|------|---------|
| 用户确认率 | ≥70% | 直接通过订单数/总订单数 |
| 有效确认率 | ≥85% | 用户确认后无需修改的比例 |
| 建单效率提升 | ≥50% | (人工建单时间 - AI建单时间)/人工建单时间 ≥ 50% |
| 错误率 | ≤2% | 生成错误模板的订单数/总订单数 |

### 3.4 规格校验规则

当订单规格与SKU规格不完全一致时，按以下规则处理：

| 情况 | 规则 | 处理方式 |
|------|------|---------|
| 规格完全一致 | 100%匹配 | 直接通过 |
| 规格主体一致，数量比例相同 | 如"500g*20" vs "1kg*10"（等量） | 标记确认，提示用户核实 |
| 规格主体一致，数量比例不同 | 如"500g*20" vs "500g*30" | 置信度降级，要求用户确认 |
| 规格完全不匹配 | 如"500g*20" vs "2.5kg*6" | 不匹配，返回未匹配 |

## 4. 测试数据集

### 4.1 数据集分类

| 数据集 | 数量 | 用途 |
|--------|------|------|
| **A. 历史订单回流测试集** | ≥50个 | 回归测试，确保迭代不破坏已有能力 |
| **B. 新客户冷启动测试集** | ≥10个客户 | 验证泛化能力 |
| **C. 边界Case测试集** | ≥20个 | 测试容错能力 |
| **D. 盲测数据集** | 20个 | 最终Benchmark，独立验收 |

### 4.2 数据集准备计划

| 数据集 | 责任人 | 时间节点 | 标注标准 |
|--------|--------|----------|----------|
| 历史订单回流 | 运营团队 | 测评启动前5个工作日 | 使用已确认的历史处理结果 |
| 新客户冷启动 | 产品团队 | 测评启动前3个工作日 | 人工标注标准答案，标注准确率≥95% |
| 边界Case | 测试团队 | 测评启动前2个工作日 | 预期结果已知，无需标注 |
| 盲测 | 独立评审 | 测评当天现场开封 | 人工标注标准答案，标注后封存 |

**数据准备要求：**
- 所有数据必须与开发团队隔离
- 盲测数据在最终验收前不得泄露给开发团队
- 标注团队需经过培训，统一标注标准

### 4.3 标注标准规范

```yaml
标注规范:
  - 标注人员: 经过培训合格后方可参与标注
  - 标注精度: 输入100%准确，输出标准答案准确率≥95%
  - 质量抽检: 随机抽取10%进行双人复核，不一致率>5%需返工
  - 标注工具: 统一使用指定标注平台
```

## 5. 评估指标体系

### 5.1 核心指标（Primary Metrics）

| 指标 | 定义 | 计算公式 | 目标 | 测量方法 |
|------|------|----------|------|----------|
| 门店匹配准确率 | 匹配正确的门店数/总门店数 | 正确匹配数÷总门店数 | ≥95% | 对比测试集标准答案 |
| SKU映射准确率 | 映射正确的商品数/总商品数 | 正确映射数÷总商品数 | ≥90% | 对比测试集标准答案 |
| 模板生成成功率 | 成功生成模板的订单数/总订单数 | 成功生成数÷总订单数 | ≥98% | 执行测试记录 |
| 用户确认率 | 无需修改直接通过的比例 | 直接通过数÷总订单数 | ≥70% | 实际使用数据统计 |

### 5.2 体验指标（Experience Metrics）

| 指标 | 定义 | 目标 | 测量方法 |
|------|------|------|----------|
| 平均处理时长 | 用户从上传到收到模板的时间 | ≤30秒 | 计时器测量P95 |
| 人工介入率 | 需要用户修改或确认的比例 | ≤30% | 实际使用数据统计 |
| 错误率 | 生成错误模板的比例 | ≤2% | 实际使用数据统计 |
| 有效确认率 | 用户确认后实际无需修改的比例 | ≥85% | 用户确认后7天内修改记录统计 |

### 5.3 业务指标（Business Metrics）

| 指标 | 定义 | 目标 | 测量方法 |
|------|------|------|----------|
| 建单效率提升 | AI建单 vs 人工建单时间差 | ≥50% | 对照实验：随机抽取20个订单，分别用AI和人工处理，计算时间差 |
| 客服咨询量下降 | 建单相关客服咨询量变化 | ≥30% | 上线前后各30天对比统计 |

### 5.4 盲测指标

| 指标 | 定义 | 目标 | 测量方法 |
|------|------|------|----------|
| 盲测准确率 | 盲测数据集上SKU映射正确率 | ≥85% | 用盲测数据集测试，对比标准答案 |
| 盲测召回率 | 盲测数据集上SKU匹配召回率 | ≥90% | 能匹配到的SKU数/总应匹配SKU数 |

**注：** 85%基准设定依据：在未知新客户数据上，AI准确率应接近人工建单水平（人工建单准确率约90-95%）。

## 6. 测试执行计划

### Phase 1：功能测试（1-2天）

**测试内容：**
1. 门店匹配 - 50个历史订单，验证Layer 0~2匹配准确率
2. SKU映射 - 所有历史订单商品，验证Layer 0~3匹配 + 规格校验
3. 规则库 - 验证字段别名映射正确性
4. 模板生成 - 31字段完整性 + 格式校验
5. 边界Case - 异常订单格式测试

**通过标准：**
- 核心指标全部达标
- 无P0/P1级bug

---

### Phase 2：性能测试（1天）

**测试内容：**
1. 响应时间 - 测量不同规模订单（1门店、10门店、50门店）的处理时间
2. 压力测试 - 模拟5个订单并发处理
3. 资源消耗 - CPU/内存监控
4. 规格校验性能 - 规格比对处理时间

**通过标准：**
- 所有性能指标达标
- 无内存泄漏
- 并发处理正常

---

### Phase 3：盲测验证（1天）

**测试内容：**
1. 用盲测数据集测试
2. 计算盲测准确率≥85%
3. 对照实验验证效率提升≥50%

**通过标准：**
- 盲测准确率≥85%
- 效率提升≥50%

---

### Phase 4：上线评审（0.5天）

**评审内容：**
1. 测试报告Review
2. 风险评估
3. 监控方案确认
4. 回滚方案确认

**通过标准：**
- 产品委员会批准
- 运维团队确认可部署

## 7. 风险评估与应对

### 7.1 风险矩阵

| 风险类别 | 风险 | 概率 | 影响 | 应对措施 |
|----------|------|------|------|----------|
| **业务风险** | SKU映射错误导致发错货 | 中 | 高 | 设置置信度<80%强制人工确认 |
| **业务风险** | 门店匹配错误导致串货 | 低 | 极高 | 货主ID二次确认 |
| **业务风险** | 规格不一致导致库存差异 | 中 | 中 | 规格校验提示 + 人工确认 |
| **技术风险** | 数据库连接池耗尽 | 低 | 高 | 连接池管理 + 超时机制 |
| **技术风险** | 大数据量OOM | 低 | 高 | 流式处理 + 内存限制 |
| **技术风险** | 并发死锁 | 低 | 高 | 事务隔离级别优化 |
| **技术风险** | 内存泄漏 | 低 | 中 | 定期重启 + 内存监控 |
| **外部依赖** | 飞书API限流 | 中 | 低 | 异步队列 + 重试机制 |
| **外部依赖** | 数据库连接失败 | 低 | 中 | 多数据源切换 + 本地缓存 |
| **数据风险** | 测试数据分布与生产不一致 | 中 | 中 | 定期更新测试数据 |
| **数据风险** | 标注错误影响评估 | 低 | 中 | 标注质量抽检机制 |
| **人员风险** | 测试人员能力不足 | 低 | 中 | 培训 + 预测试演练 |
| **人员风险** | 标注标准不一致 | 中 | 中 | 统一培训 + 标注规范文档 |

### 7.2 监控告警方案

| 监控项 | 阈值 | 告警方式 |
|--------|------|---------|
| SKU映射成功率 | <90% | 短信告警 |
| 有效确认率 | <85% | 邮件告警 |
| 平均处理时间 | >30秒 | 邮件告警 |
| 错误率 | >5% | 钉钉告警 |
| 置信度异常分布 | 大量<80% | 飞书通知 |
| 数据库连接失败 | 连续3次 | 短信+电话 |
| 内存占用 | >80% | 告警 |
| API限流 | 触发限流 | 记录+通知 |
