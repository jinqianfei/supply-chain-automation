## 1. 概述

### 1.1 目的

本方法论为AI建单助手Skill的测评提供通用框架，适用于所有Skill的验收测试。目标是：

1. 建立统一的测评标准和方法
2. 确保测评结果可量化、可复现
3. 指导Skill从开发到上线的质量把控

### 1.2 适用范围

- 所有AI建单助手Skill
- 可扩展至其他AI Agent的测评

### 1.3 核心原则

| 原则 | 说明 | 应用 |
|------|------|------|
| **结果导向** | 测评的是Skill的输出质量，而非实现细节 | 关注准确率、召回率等结果指标 |
| **数据驱动** | 用真实数据验证，用数据说话 | 建立标准化数据集 |
| **风险前置** | 在上线前识别风险，而非上线后救火 | 风险矩阵 + 监控方案 |
| **持续迭代** | 测评不是一次性活动，是持续过程 | 上线后跟踪计划 |

## 3. 验收标准设计

### 3.1 三层指标体系

所有Skill的测评指标都应该分为三层：

| 层次 | 回答问题 | 通用指标 |
|------|---------|---------|
| **核心层** | 功能对不对？ | 准确率、召回率、成功率 |
| **体验层** | 用起来爽不爽？ | 响应时间、人工介入率 |
| **业务层** | 真的创造价值了吗？ | 效率提升、错误率下降 |

### 3.2 指标设计原则

| 原则 | 说明 | 反例 |
|------|------|------|
| 可量化 | 指标必须有明确数值 | "效果好不好" → 准确率≥90% |
| 可测量 | 有明确测量方法 | "用户体验好" → 平均处理时长≤30秒 |
| 有目标 | 目标值有挑战但可达成 | "做到最好" → 用户确认率≥70% |
| 有业务含义 | 指标反映业务价值 | "调用次数多" → 人工介入率≤30% |

### 3.3 通用验收标准模板

```yaml
# Skill验收标准模板
skill_name: "Skill名称"
version: "版本号"

acceptance_criteria:
  functional:
    accuracy: ≥ 目标值    # 如准确率≥90%
    recall: ≥ 目标值      # 如召回率≥85%
    success_rate: ≥ 目标值 # 如成功率≥95%
  
  performance:
    response_time: ≤ N秒  # 单次处理时间 P95
    latency: ≤ Nms       # 单次操作延迟
    memory: ≤ NMB         # 内存占用
  
  business:
    efficiency_improvement: ≥ N%  # 效率提升
    error_rate: ≤ N%              # 错误率
    effective_confirmation_rate: ≥ N%  # 有效确认率

  experience:
    user_confirmation_rate: ≥ N% # 用户直接通过率
    manual_intervention_rate: ≤ N% # 人工介入率
```

### 3.4 指标量化要求

所有指标必须满足SMART原则：

| 要求 | 说明 | 示例 |
|------|------|------|
| **S - Specific（具体）** | 指标定义清晰，无歧义 | "准确率≥90%"而非"准确率高" |
| **M - Measurable（可测量）** | 有明确的测量方法和计算公式 | 准确率 = 正确数/总数 |
| **A - Achievable（可达成）** | 目标有挑战但可达成 | 90%可行，100%不现实 |
| **R - Relevant（相关）** | 指标与业务目标相关 | 准确率直接影响用户体验 |
| **T - Time-bound（有时限）** | 有明确的测量时间范围 | 取P95而非平均值 |

## 4. 测试数据集设计

### 4.1 四类数据设计原则

| 数据类别 | 目的 | 设计原则 |
|----------|------|----------|
| **历史回流数据** | 回归测试，确保不破坏已有能力 | 用过去6个月的真实数据，已有正确答案 |
| **冷启动数据** | 验证泛化能力 | 用未曾见过的新客户/新场景数据 |
| **边界Case数据** | 测试容错能力 | 构造异常输入（缺字段、错误格式、极端值） |
| **盲测数据** | 验证真实泛化能力 | 与开发过程隔离，仅最终验收时使用 |

### 4.2 数据集规模指南

| 数据集 | 最小规模 | 说明 |
|--------|---------|------|
| 历史回流 | ≥50条 | 覆盖常见场景，有统计意义 |
| 冷启动 | ≥10个客户/场景 | 验证新场景泛化 |
| 边界Case | ≥20条 | 覆盖主要异常类型 |
| 盲测 | ≥20条 | 与开发数据量相当，保证公平性 |

### 4.3 数据集准备计划

| 阶段 | 任务 | 责任人 | 时间节点 |
|------|------|--------|----------|
| 数据收集 | 收集原始数据 | 运营/产品团队 | T+N日 |
| 数据清洗 | 清洗无效数据 | 测试团队 | T+N+1日 |
| 数据标注 | 标注标准答案 | 经过培训的标注团队 | T+N+2日 |
| 质量抽检 | 随机10%双人复核 | 质量负责人 | T+N+3日 |
| 盲测封存 | 盲测数据密封保存 | 独立评审 | 测评当天 |

### 4.4 数据标注规范

```yaml
标注规范:
  标注人员:
    - 经过培训合格后方可参与标注
    - 标注准确率≥95%方可正式标注
    
  标注质量:
    - 随机抽取10%进行双人复核
    - 不一致率>5%需返工
    
  标注标准:
    - 统一使用指定标注平台
    - 遵循标注规范文档
```

## 5. 测试执行方法

### 5.1 分层测试法

| 测试层 | 目的 | 执行时机 | 执行主体 |
|--------|------|---------|---------|
| 单元测试 | 验证最小单位功能 | 开发过程 | 开发者 |
| 集成测试 | 验证模块间协作 | 开发完成 | 测试团队 |
| 系统测试 | 验证完整业务流程 | 上线前 | 测试团队 |
| 盲测 | 验证真实泛化能力 | 最终验收 | 独立评审 |

### 5.2 回归测试策略

每次Skill更新后，必须执行：

1. **历史回流测试** - 确保已知问题不复发
2. **增量测试** - 针对新功能的测试
3. **全量测试** - 完整测试集（定期执行）

### 5.3 异常测试法

| 异常类型 | 测试方法 | 预期结果 |
|---------|---------|---------|
| 输入为空 | 传入空数据 | 明确错误提示，不崩溃 |
| 格式错误 | 传入错误格式 | 识别错误，返回合理提示 |
| 极端值 | 传入边界值（0、最大值） | 正确处理，不崩溃 |
| 超时 | 模拟响应慢 | 超时处理，不卡死 |
| 内存泄漏 | 大数据量处理 | 内存占用稳定 |
| 并发死锁 | 多线程并发请求 | 正确处理，无死锁 |

## 6. 性能评估方法

### 6.1 性能指标定义

| 指标 | 定义 | 测量方法 |
|------|------|---------|
| 响应时间 | 从输入到输出的总时间 | 计时器，取P95/P99 |
| 操作延迟 | 单次操作的响应时间 | 数据库/缓存监控 |
| 吞吐量 | 单位时间处理能力 | 处理数÷时间 |
| 资源占用 | CPU/内存使用量 | 监控工具 |
| 并发能力 | 同时处理的最大请求数 | 压力测试 |

### 6.2 性能测试类型

| 类型 | 目的 | 方法 |
|------|------|------|
| 负载测试 | 验证正常负载下的性能 | 逐步增加请求，找到最优负载点 |
| 压力测试 | 验证极限负载下的稳定性 | 超过最优负载直到性能下降 |
| 耐久性测试 | 验证长时间运行的稳定性 | 持续运行N小时，监控资源泄漏 |

### 6.3 性能目标设定原则

| 原则 | 说明 | 示例 |
|------|------|------|
| 有用户视角 | 从用户感受出发设定目标 | 30秒是用户等待极限 |
| 有统计意义 | 用P95/P99而非平均值 | P95≤10秒 |
| 有业务边界 | 考虑业务场景的限制 | 建单类任务≤30秒 |
| 有优化空间 | 目标有一定挑战性 | 当前5秒，目标3秒 |

## 7. 风险评估方法

### 7.1 风险识别框架

| 风险来源 | 典型风险 | 评估维度 |
|----------|---------|---------|
| 技术风险 | 系统超时、内存泄漏、连接池耗尽、OOM、并发死锁 | 影响程度×发生概率 |
| 数据风险 | 数据质量差、覆盖不足、标注错误、分布不一致 | 影响范围×修复难度 |
| 业务风险 | 输出错误、错误率高、串货发错 | 业务损失×发生频率 |
| 外部风险 | 依赖服务不可用、API限流、第三方服务故障 | 影响程度×依赖程度 |
| 人员风险 | 测试人员能力不足、标注标准不一致 | 项目延期×质量影响 |

### 7.2 风险矩阵

|  | 低影响 | 中影响 | 高影响 |
|--|--------|--------|--------|
| **高概率** | 监控 | 应对 | 优先解决 |
| **中概率** | 观察 | 监控 | 应对 |
| **低概率** | 接受 | 观察 | 监控 |

### 7.3 风险应对策略

| 策略 | 适用情况 | 具体措施 |
|------|---------|---------|
| 避免 | 风险高且概率高 | 修改设计，消除风险 |
| 减轻 | 风险高但概率低 | 加防护机制，降低影响 |
| 转移 | 风险中且概率中 | 人工复核、保险 |
| 接受 | 风险低且概率低 | 监控，接受小概率问题 |

### 7.4 风险监控清单

| 风险类别 | 监控指标 | 阈值 |
|----------|---------|------|
| 技术风险 | 内存使用率 | >80%告警 |
| 技术风险 | 数据库连接数 | >80%池容量告警 |
| 技术风险 | 错误率 | >5%告警 |
| 业务风险 | 有效确认率 | <85%告警 |
| 业务风险 | 效率提升 | <50%达标 |
| 数据风险 | 测试数据与生产偏差 | 定期检测 |

## 8. 规格校验与边界处理

### 8.1 规格校验原则

当输入规格与系统规格不完全一致时，按以下规则处理：

| 情况 | 规则 | 处理方式 |
|------|------|---------|
| 规格完全一致 | 100%匹配 | 直接通过 |
| 规格主体一致，数量成比例 | 如"500g*20" vs "1kg*10" | 提示用户核实 |
| 规格主体一致，数量不成比例 | 如"500g*20" vs "500g*30" | 置信度降级，人工确认 |
| 规格完全不匹配 | 如"500g" vs "2.5kg" | 不匹配，返回未匹配 |

### 8.2 模板格式校验

| 校验项 | 校验规则 | 不通过处理 |
|--------|---------|-----------|
| 日期格式 | YYYY-MM-DD | 提示格式错误 |
| 数量范围 | >0 | 提示数量异常 |
| 必填字段 | 非空 | 拒绝生成，提示缺失 |
| 字段长度 | 符合数据库限制 | 截断或提示超长 |

## 9. 监控与跟踪方法

### 9.1 监控体系设计

| 监控类型 | 监控内容 | 阈值设定 |
|----------|---------|---------|
| 系统监控 | CPU/内存/磁盘 | 系统阈值 |
| 应用监控 | 请求量/错误率 | 基线±Nσ |
| 业务监控 | 核心指标达成率 | 验收标准阈值 |
| 告警监控 | 异常事件 | 第一时间告警 |

### 9.2 上线后跟踪机制

| 阶段 | 跟踪频率 | 跟踪内容 |
|------|---------|---------|
| 观察期（上线后1-3天） | 每小时 | 核心指标 + 系统状态 |
| 稳定期（上线后1周） | 每日 | 指标趋势 + 问题汇总 |
| 运营期（上线后1月+） | 每周 | 业务价值 + 优化建议 |
| 评估期（上线后3月+） | 每月 | 规则库版本评估，更新测试集 |

## 10. Go/No-Go决策方法

### 10.1 决策原则

| 原则 | 说明 | 应用 |
|------|------|------|
| **全部通过才能上线** | 没有"差不多就行" | 所有Checklist项必须打勾 |
| **一票否决** | 重大问题直接否决 | P0级Bug未关闭 → 否决 |
| **量化决策** | 用数据说话 | 准确率85% vs 目标90% → 不能上线 |
| **可追溯** | 决策有据可查 | 记录未关闭项和例外情况 |

## 11. 应用指南

### 11.1 新Skill测评流程

```
Step 1: 确定测评范围
   └─ 识别Skill核心功能
   └─ 定义测评边界

Step 2: 制定验收标准
   └─ 设计三层指标体系
   └─ 设定量化目标
   └─ 定义测量方法

Step 3: 准备测试数据
   └─ 收集/构造四类数据
   └─ 标注标准答案
   └─ 执行数据准备计划

Step 4: 执行测试
   └─ 功能测试
   └─ 性能测试
   └─ 盲测验证

Step 5: 风险评估
   └─ 识别风险（技术/业务/数据/外部）
   └─ 制定应对措施

Step 6: 上线决策
   └─ Go/No-Go Checklist
   └─ 决策输出

Step 7: 上线后跟踪
   └─ 监控核心指标
   └─ 持续优化
```

### 11.2 常见误区与应对

| 误区 | 问题 | 应对 |
|------|------|------|
| 只测正常路径 | 边界情况全部崩溃 | 必须有边界Case测试集 |
| 用训练数据测试 | 过拟合，无法泛化 | 必须有盲测数据集 |
| 目标设100% | 自我欺骗 | 目标要有挑战但可达成 |
| 上线后不管 | 问题暴露晚 | 必须有上线后跟踪计划 |
| 凭感觉判断 | 无标准、无追溯 | 用量化指标和数据决策 |
| 指标不可测量 | 无法验收 | 所有指标必须有量化定义和测量方法 |
| 数据准备不足 | 测试无法执行 | 必须有数据集准备计划 |

### 11.3 责任矩阵

| 任务 | 责任人 | 审核人 |
|------|--------|--------|
| 数据收集 | 运营团队 | 产品团队 |
| 数据标注 | 标注团队 | 质量负责人 |
| 测试执行 | 测试团队 | 技术负责人 |
| 风险评估 | 技术+产品 | 产品委员会 |
| 上线决策 | 产品委员会 | CTO/业务负责人 |
| 上线后跟踪 | 运营+产品 | 产品负责人 |
