## 1. 项目背景

### 1.1 业务场景

**订单建单**是供应链的核心环节：客户发来订单（Excel/图片/PDF/文字）→ 人工解析 → 匹配SKU → 生成标准出库单 → 交付仓库。

**痛点**：每天处理 50-100 个订单，人工耗时 2-3 小时，且容易出错。

### 1.2 项目目标

| 目标 | 具体指标 |
|------|----------|
| 自动化 | 订单解析自动化率 ≥ 90% |
| 准确率 | SKU匹配准确率 ≥ 95% |
| 效率 | 单订单处理时长 < 5 秒 |
| 可靠性 | 7×24 小时稳定运行 |

## 3. 技术选型

### 3.1 为什么用 Skill 架构？

| 考量 | 选项 A：LangChain | 选项 B：Skill（我们选）|
|------|-------------------|------------------------|
| 定制性 | 中 | 高 |
| 维护成本 | 高 | 低 |
| 调试友好 | 低 | 高 |
| 企业适配 | 需改造 | 原生支持 |
| 扩展性 | 依赖生态 | 完全自主 |

**结论**：Skill 架构更适合「特定业务场景 + 需深度定制」的企业AI应用。

### 3.2 为什么用 PostgreSQL？

| 考量 | MongoDB | MySQL | PostgreSQL（我们选）|
|------|---------|-------|---------------------|
| JSON 支持 | 好 | 一般 | 好 |
| 向量搜索 | 需插件 | 无 | pgvector |
| 复杂查询 | 一般 | 一般 | 好 |
| 生态 | 一般 | 好 | 好 |

**结论**：PostgreSQL 一库多用，兼顾结构化查询和 JSON 扩展。

### 3.3 为什么用 OpenClaw 作为 Agent 框架？

| 考量 | 选型理由 |
|------|----------|
| 多渠道 | 同时支持飞书/钉钉/微信/Telegram |
| MCP 生态 | 内置大量工具（feishu_doc / feishu_bitable / 供应链工具等）|
| 本地部署 | 数据不出企业，安全可控 |
| 定制化 | 高度可配置，适合企业场景 |

## 4. 痛点分析与解决方案

### 4.1 痛点 1：门店匹配准确率低

**问题**：客户订单的门店名经常是简称、别名、拼写错误。

**解决**：5 层匹配算法

| 层 | 算法 | 置信度 |
|----|------|--------|
| L0 | 别名表查表 | 0.98 |
| L1 | 精确匹配 | 0.95 |
| L2 | 模糊匹配（LIKE %name%）| 0.80 |
| L3 | 分词关键词匹配 | 0.70 |
| L4 | 兜底（返回未匹配，供人工确认）| 0.00 |

### 4.2 痛点 2：SKU 映射效率低

**问题**：每次订单都要查 1832 条 SKU，响应慢。

**解决**：
1. 预加载 SKU 到内存（启动时一次）
2. 索引优化（按货主ID分区）
3. 缓存热点 SKU

### 4.3 痛点 3：版本管理混乱

**问题**：代码迭代多次后，不知道当前是什么版本。

**解决**：三处一致校验

```
VERSION 文件 + CHANGELOG.md + SKILL.md 头
每次启动自动校验，不一致则阻断
```

### 4.4 痛点 4：记忆系统脆弱

**问题**：会话记录容易断档，重要决策找不到。

**解决**：5 层记忆架构

```
L1 触发层（自动化守护）
L2 协议层（SESSION_START/END）
L3 索引层（memory_search）
L4 真相源（git/代码/mtime）
L5 决策层（MEMORY.md 手账）
```

## 6. 经验总结

### 6.1 关键经验

1. **先跑通，再优化** — 不要一开始就追求完美，先让流程跑起来
2. **数据是护城河** — SKU/门店/别名数据越丰富，匹配越准
3. **自动化守护** — 靠人自觉必失败，必须用脚本 + launchd
4. **版本管理** — 三处一致校验，避免"我不知道现在是什么版本"

### 6.2 踩过的坑

| 坑 | 代价 | 解决方案 |
|----|------|----------|
| MEMORY.md 当真相源 | 被打脸"你不是说5.8吗" | git/代码/mtime 才是真相 |
| 断档 4 天 | 决策丢失 | check_continuity.sh |
| AI 跳过主入口 | 绕过技术护栏 | __getattribute__ 锁死 |
| 一次改太多 | 难以回滚 | 一次只改一类问题 |
