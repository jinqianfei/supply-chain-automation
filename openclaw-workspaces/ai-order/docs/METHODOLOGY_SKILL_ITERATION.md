# 方法论：如何通过"测评"实现 Skill 持续迭代

> **作者**：AI建单助手
> **日期**：2026-06-08
> **案例**：skill_order_to_huading_template v5.0 → v5.9.0
> **核心观点**：**没有测评的迭代是赌博**

---

## 0. 一句话总结

> **"评测 = 拉尺子；迭代 = 改尺子"** — 评测告诉你哪里偏了，迭代把偏的部分改回来，循环往复。

---

## 1. 为什么要"测评"驱动迭代？

### 1.1 没有测评的迭代（反面案例）

| 症状 | 后果 |
|------|------|
| 改代码后凭"我觉得好了"上线 | 实际把旧的 80% 改坏到 60%，但没人发现 |
| 老板/用户反馈"好像变慢了" | 不知道慢在哪一环，没法定位 |
| 半年后想"上次我们改了什么" | 翻 git log 找不到，记忆也丢了 |
| 新人接手 | 不知道哪部分代码是稳定的，哪部分是高风险区 |

### 1.2 有测评的迭代（正面案例）

| 症状 | 好处 |
|------|------|
| 改代码前跑 baseline 评测 | 知道起点在哪 |
| 改完跑同一份评测 | 客观判断进步/退步（**数据说了算，不是人说了算**）|
| 评测结果写入 CHANGELOG | 每次发布都有证据 |
| 失败时回滚到上一个"评测全过"的版本 | 安全感 |

---

## 2. 测评驱动迭代的 5 步法（**AI 建单助手案例**）

```
┌────────────┐    ┌────────────┐    ┌────────────┐    ┌────────────┐    ┌────────────┐
│ 1️⃣ 建尺子  │ →  │ 2️⃣ 评起点  │ →  │ 3️⃣ 找问题  │ →  │ 4️⃣ 迭代   │ →  │ 5️⃣ 再评    │ → 循环
└────────────┘    └────────────┘    └────────────┘    └────────────┘    └────────────┘
```

### 步骤 1️⃣ 建尺子（设计测评指标）

**目标**：把"好"变成可量化的数字。

**AI 建单助手案例**：

| 维度 | 指标 | 怎么测 |
|------|------|--------|
| 功能正确性 | 门店匹配准确率 | 输入 50 个真实门店名，看匹配对几个 |
| 功能正确性 | SKU 匹配准确率 | 输入 100 个真实商品名，看匹配对几个 |
| 性能 | 单订单处理时长 | 测 10 次取中位数 |
| 性能 | 数据库查询次数 | log SQL，统计 |
| 鲁棒性 | 异常订单不崩溃 | 故意给错误输入，看是否友好报错 |
| 用户体验 | 必须用户确认 | 100% 触发 need_confirm，不允许自动确认 |
| 用户体验 | 置信度 < 80% 告警 | 故意给模糊商品，看是否标红 |
| 可维护性 | 内部函数不暴露 | AI 不能跳过 execute() 直接调 _match_store |

**原则**：
- ✅ 指标**可量化**（数字/百分比/退出码）
- ✅ 指标**可重复**（同输入得同结果）
- ✅ 指标**覆盖**主要场景 + 边界场景
- ❌ 不要"代码风格好不好"这种主观项

### 步骤 2️⃣ 评起点（Baseline 测评）

**目标**：在改任何代码前，先知道现在的真实水平。

**AI 建单助手案例（2026-06-04 测评）**：
- 测评文件：`SKILL_EVALUATION_REPORT.md`
- 测评数据：product_sku=1832 条、store_list=3327 条、别名表=30 条
- 测评结果：**v5.8 五星评分（5/5）**
- 测评结论：所有模块通过，但**未接 CI，每次改动靠人工跑**

**关键发现**：
- 5 个模块都"过了"，但**没有自动化** → 下次改动时再跑一遍测评
- 这是"一次性测评"，不是"持续测评"

### 步骤 3️⃣ 找问题（信号检测）

**目标**：从测评数据里挖出真实的"痛点"。

**AI 建单助手案例**：

| 信号 | 来源 | 真实问题 |
|------|------|----------|
| 老板问"你之前说 5.8，实际是 5.9" | MEMORY.md 错乱 | **真相源混乱** — 不是代码问题，是记忆问题 |
| 6-4~6-7 断档 4 天 | SESSION_END 未执行 | **没有自动化守护** — 靠人自觉必失败 |
| 1/5 模块"已删除/已合并" | 评测没更新 | **测评与代码脱节** — 5 月合并数据库，但测评还在测旧表 |
| 评测发现 SKU 78% 是低匹配 | 测试数据 | **算法兜底不力** — 模糊匹配兜底太宽松 |
| AI 多次跳过 execute() 主入口 | 看 git diff | **没有技术护栏** — AI 自由度过大 |

**SignalDetector 三层**：
```
L1 错误信号   → 测试失败 / 报错 / 崩溃
L2 性能信号   → 慢 / 卡 / 超时 / 资源占用
L3 体验信号   → 用户抱怨 / 自动确认 / 数据错乱
```

### 步骤 4️⃣ 迭代（按优先级改）

**目标**：用最小代价解决最痛的问题。

**AI 建单助手案例（优先级排序）**：

| 优先级 | 问题 | 修复 | 验证 |
|--------|------|------|------|
| P0 | MEMORY.md 当真相源 | version_check.sh 三处一致 | bash test |
| P0 | AI 跳主入口 | __getattribute__ 技术锁 | 单元测试 |
| P1 | 无自动化守护 | 3 个脚本 + launchd | 14 项测试 |
| P1 | 评测与代码脱节 | 重写 schema.sql，删旧表 | DB 查询 |
| P2 | 模糊匹配兜底松 | 加 product_name_alias 精确表 | 评测 95%→98% |
| P3 | SKU 78% 告警多 | Phase 2 用学习数据自动提别名 | 待 Phase 2 数据 |

**原则**：
- ✅ **一次只改一类问题**（P0 改完才改 P1）
- ✅ **改完立即跑同一份评测**
- ✅ **退步就回滚**（不冒险）
- ❌ 不要"改 A 时顺手改 B"（污染 baseline）

### 步骤 5️⃣ 再评（确认进步）

**目标**：用数据证明这次迭代有效。

**AI 建单助手案例**：

| 指标 | v5.8 baseline | v5.9.0 迭代后 | 变化 |
|------|---------------|---------------|------|
| 门店匹配率 | 100% (3/3 测试) | 100% | 持平 |
| SKU 匹配率 | 100% (3/3 测试) | 100% | 持平 |
| 内部函数可绕过 | ❌ 容易 | ✅ 锁死 | **+质变** |
| 版本号一致性 | ❌ 错乱 | ✅ 自动校验 | **+质变** |
| 日志断档 | ❌ 4 天 | ✅ 24h 自动告警 | **+质变** |
| 评测自动化 | ❌ 手工 | ✅ 14 项测试脚本 | **+质变** |

**关键**：
- 表面看"匹配率"没提升，**实际上"质变"在不该出错的地方不再出错了**
- 这就是"持续迭代"和"一次性大改"的区别：**持续迭代解决的是"过程可靠性"问题**

## 3. 4 个反模式（避坑）

### 反模式 1：把"跑通"当"评测"

```diff
- "我跑了，输出对，就过了"
+ "我跑了测试，4/4 通过；又跑了边界场景，2/2 通过；又跑了负面场景，3/3 友好报错"
```

### 反模式 2：改完不发版本

```diff
- 改完直接用，不写 CHANGELOG
+ 改完 → VERSION+1 → CHANGELOG 加条目 → git commit → tag
```

### 反模式 3：把 MEMORY.md 当事实

```diff
- "我记得上次说 5.8"
+ "让我查 VERSION 文件 / git log / 文件 mtime 再答"
```

### 反模式 4：一次迭代改太多

```diff
- "这次顺便把 X、Y、Z 都改了"
+ "这次只解决 P0 问题；下次迭代再处理 P1"
```

## 4. 持续迭代的"飞轮"

```
   评测数据
      ↓
   发现问题
      ↓
   优先级排序
      ↓
   小步快跑（一次一类）
      ↓
   同评测再跑 → 看变化
      ↓
   写入版本 + commit
      ↓
   （回到"评测数据"）
```

**飞轮要点**：
- 每转一圈，**Skill 变得更可靠一点点**
- 飞轮不会自己转，**需要人/AI 持续投入**
- 飞轮的关键是"评测"环节，**没有它就只是瞎改**

## 5. 测评自动化的 3 个阶段

| 阶段 | 自动化程度 | 工具 | 投入 |
|------|-----------|------|------|
| **L1 手工** | 跑测试靠人 | `python test_xxx.py` | 0 |
| **L2 脚本** | 一键全跑 | `bash run_all_tests.sh` | 1h |
| **L3 守护** | 提交即跑 / 定时跑 | CI / cron / launchd | 1d |

## 6. 关键学习（来自 v5.8 → v5.9.0 真实迭代）

### 学习 1：评测要包含"反向测试"

```python
def test_ai_cannot_bypass_main_entry():
    """AI 跳主入口应当被拦截"""
    skill = OrderToHuadingTemplate(...)
    with pytest.raises(ForbiddenDirectCall):
        skill._match_sku(...)  # 应该失败
```

不是"功能跑通"就行，还要测"不该发生的没发生"。

### 学习 2：版本号一致性是 1 分钟可解决的问题

- 修：`bash scripts/version_check.sh`
- 跑：自动核对 VERSION/CHANGELOG/SKILL.md 三处
- 价值：避免 1 个月后再被打脸"你不是说 5.8 吗"

### 学习 3：记忆和代码一样要"测"

- 测版本号：version_check.sh
- 测断档：check_continuity.sh
- 测新鲜度：startup_check.py 里的 memory_fresh

**记忆不是"写下来就行"，是"每天/每次启动都验证"**。

### 学习 4：迭代节奏

| 频率 | 适用 | 案例 |
|------|------|------|
| 每天 | 修 bug / 微调 | daily_wrap 修字段 |
| 每周 | 加小功能 | Phase 1 加事件总线 |
| 每月 | 架构调整 | Phase 3 智能归档 |
| 季度 | 大版本 | v5.0 → v6.0 |

**不要每天做架构调整**（容易烂），**不要季度才修 bug**（积累成灾难）。

---

## 7. 一页纸总结

> **AI Skill 持续迭代 = 评测 + 找问题 + 小步改 + 再评测**
>
> **评测指标要可量化、可重复、覆盖主场景+边界场景**
>
> **迭代要小步快跑、一次一类、退步就回滚**
>
> **记忆和代码一样要"测"**
>
> **飞轮不转就要主动推**

---

## 8. 自检问题（每个迭代周期问自己 4 个问题）

1. **这次迭代前我跑过 baseline 评测吗？**
2. **这次迭代我只改了一类问题吗？**
3. **这次迭代后我跑过同一份评测吗？**
4. **这次迭代的证据（git commit + 评测结果 + CHANGELOG）齐全吗？**

任何一题答"否" → 这次迭代**不算**。
