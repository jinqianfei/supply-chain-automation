## 3. 飞轮（**持续迭代的核心机制**）

### 3.1 飞轮结构

```
       ┌─────── 飞轮入口 ───────┐
       ↓                         ↑
  收集反馈                       部署上线
  (Phase 1 已完成)              (新版本)
       ↓                         ↑
  发现问题（SignalDetector）    再评测
  (Phase 2 已完成)              (CI 跑全套)
       ↓                         ↑
  优先级排序                    确认进步/回滚
  (本月已建立)                  (CHANGELOG)
       ↓                         ↑
  实施迭代                      记录决策
  (P0/P1/P2)                    (memory/2026-XX-XX.md)
       └─────── 飞轮出口 ───────┘
```

### 3.2 每圈飞轮的最小动作

| 步骤 | 输出 | 时间预算 |
|------|------|----------|
| 1. 看 1 份反馈 | 1 条新 insight | 5min |
| 2. 找 1 个问题 | 1 个明确 bug | 10min |
| 3. 改 1 处代码 | 1 个 commit | 30min |
| 4. 跑 1 遍评测 | 1 份 PASS/FAIL | 5min |
| 5. 写 1 行 CHANGELOG | 版本 +1 | 2min |
| **小计** | | **~1h** |

### 3.3 飞轮失败模式

| 失败 | 现象 | 修复 |
|------|------|------|
| 反馈没人看 | 事件采集了但无 insight | 每周一次反馈 review |
| 问题找不到 | 数据有但不会挖 | 用 `v_layer_success_rate` 视图找低分项 |
| 改了不评 | 上线后才发现退步 | git pre-commit hook 强制跑测试 |
| 评了不记 | 评测结果没归档 | 写入 `docs/eval_history/` |
| 记了不用 | 历史数据没价值 | 每月一次趋势分析 |

### 7.2 每周检查

```sql
-- 看层成功率
SELECT layer_name, total_attempts, success_count, success_rate
FROM layer_success_rate
WHERE total_attempts > 0
ORDER BY success_rate ASC;
```

**告警规则**：
- 任何层 success_rate < 70% → 飞书告警
- 周环比下降 > 5% → 飞书告警

```sql

SELECT * FROM v_daily_feedback_stats
WHERE order_date >= CURRENT_DATE - 30
ORDER BY order_date DESC;
```
