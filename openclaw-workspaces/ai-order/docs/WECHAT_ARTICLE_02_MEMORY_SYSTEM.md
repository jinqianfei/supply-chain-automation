## 0. 开篇语

很多AI助手用着用着就"失忆"了——上次讨论了什么？上次改了什么？为什么又犯了同样的错？

**AI的记忆，不是写下来就行，是要每天/每次启动都验证。**

本文是「AI建单助手」记忆系统的完整方法论，涵盖：5层架构、4W质量标准、防断档机制、以及可落地的维护脚本。

---

## 1. 为什么需要记忆系统？

### 1.1 没有记忆的AI

| 症状 | 后果 |
|------|------|
| 每次问"上次我们改了什么" | 回答"我不知道" |
| 决策找不到依据 | 重复讨论同样的问题 |
| 断档 4 天 | 只能事后追溯，无法预防 |
| 新人接手 | 不知道哪些决策是"定稿"，哪些是"草稿" |

### 1.2 有记忆的AI

| 症状 | 好处 |
|------|------|
| 每次启动自检 | 知道当前版本、状态、待办 |
| 每日 10:00 日结 | 昨天发生了什么，一清二楚 |
| 决策带证据 | 回溯时有据可查 |
| 4W 质量标准 | 每条记忆可验证、可追溯 |

## 2. 5 层记忆架构

```
┌──────────────────────────────────────────────────────────┐
│ L5  决策层：MEMORY.md（人类可读，7 天摘要）               │
├──────────────────────────────────────────────────────────┤
│ L4 真相源：git log + 代码 + 文件 mtime（不可篡改）       │
├──────────────────────────────────────────────────────────┤
│ L3 索引层：memory_search 语义检索 + 项目目录树          │
├──────────────────────────────────────────────────────────┤
│ L2 协议层：SESSION_START / SESSION_END / PENDING 协议     │
├──────────────────────────────────────────────────────────┤
│ L1 触发层：version_check + 每日 10:00 日结 + Phase 1     │
└──────────────────────────────────────────────────────────┘
```

### 2.1 L1 触发层（自动化守护）

| 触发 | 工具 | 频率 | 失败处理 |
|------|------|------|---------|
| 启动 Skill | `version_check.sh` | 每次启动 | 阻断任务，要求修复 |
| 每日 10:00 | `daily_wrap.sh` | 每天 | 飞书提醒 + 写报告 |
| 订单完成 | `order_complete` 事件 | 每次 | 写入反馈表 |
| 启动 4 项 | `startup_check.py` | 每次启动 | 警告，不阻断 |
| 断档 > 24h | `check_continuity.sh` | 每日 | 飞书 P0 告警 |

### 2.2 L2 协议层（流程约束）

```
memory/
├── SESSION_START_PROTOCOL.md    ← 启动协议（4 项自检）
├── SESSION_END_PROTOCOL.md      ← 结束协议（6 步）
├── PENDING_PROTOCOL.md          ← 未完成事项追踪
└── MEMORY.md                    ← 真相摘要（手账）
```

### 2.3 L3 索引层（主动召回）

- memory_search 语义检索
- 项目目录树（skills/ + docs/ + memory/）
- 自动 reindex（103 文件 / 8650 关键词）

### 2.4 L4 真相源（不可篡改）

- git log（commit sha + message）
- 代码（VERSION / CHANGELOG / SKILL.md）
- 文件 mtime（最后修改时间）

**核心铁律**：git/代码/mtime 是真相源，MEMORY.md 是手账。

### 2.5 L5 决策层（人类可读）

- 最近 7 天会话摘要
- 待办事项（PENDING.md）
- 关键决策记录

## 3. 4W 质量标准

每条记忆必须带 4W：

| W | 含义 | 缺失后果 |
|---|------|----------|
| **When** | 绝对日期 + 相对时间 | 不知道"什么时候的事" |
| **What** | 客观事实 | 不知道"发生了什么" |
| **Why** | 触发原因 | 不知道"为什么" |
| **Witness** | 证据（git sha / 文件 / 数据）| 无法验证真假 |

### 3.1 质量分计算

```
quality = (has_when + has_what + has_why + has_witness) / 4
       × (verified_recently ? 1.0 : 0.5)
       × (has_evidence ? 1.0 : 0.7)
```

### 3.2 分档标准

| 质量分 | 等级 | 处理方式 |
|--------|------|----------|
| ≥ 0.8 | ✅ 高 | 保持 |
| 0.5-0.8 | ⚠️ 中 | 补充 |
| < 0.5 | ❌ 低 | 必须补全 |

### 3.3 自动检查

```bash
python3 scripts/check_memory_quality.py --strict
```

当前状态：3 ✅ 高 / 3 ⚠️ 中 / 0 ❌ 低

## 4. 防断档机制

### 4.1 断档检测

| 状态 | 定义 | 动作 |
|------|------|------|
| OK | 最新日志 < 24h | 正常 |
| WARN | 24-72h | 飞书告警 |
| P0 | > 72h | 飞书 P0 告警 + 阻断启动 |

### 4.2 强制日结

- **时间**：每日 10:00（由老板指定）
- **内容**：总结昨天数据
- **输出**：`/tmp/daily_wrap_YYYY-MM-DD.md`
- **调度**：launchd（macOS 原生）

### 4.3 启动 4 项自检

每次启动 Skill 时自动检查：

| # | 检查项 | 失败处理 |
|---|--------|----------|
| 1 | version_check（三处一致）| 阻断任务 |
| 2 | git_clean（无未提交重要修改）| 警告 |
| 3 | memory_fresh（MEMORY.md < 7 天）| 警告 |
| 4 | no_pending（无紧急项超 24h）| 警告 |

### 4.4 周归档

- 14+ 天日志自动移到 `archive/`
- launchd 每周日 03:30 跑

### 4.5 月复盘

- 每月 1 日 04:00 跑
- 生成月度趋势报告
- 30+ 天未更新项标记

## 5. 决策记录模板

每次重要决策必须记录：

```markdown
## 决策：[决策名]（YYYY-MM-DD）
**触发**：[什么事件]
**方案对比**：A / B / C（选定）
**理由**：[为什么]
**反例/失败条件**：[什么情况下回滚]
**批准人**：[金姐]
**证据**：git commit / 评测结果 / 文件
**下次复审**：[YYYY-MM-DD]
```

## 7. 一页纸总结

> 1. **5 层架构**：L1 触发 / L2 协议 / L3 索引 / L4 真相 / L5 决策
> 2. **4W 标准**：每条记忆带 When/What/Why/Witness + 质量分（0-1）
> 3. **防断档**：每日 10:00 日结 + 24h 断档告警 + 4 项启动自检
> 4. **核心铁律**：git/代码/mtime 是真相源，MEMORY.md 是手账
> 5. **自动化维护**：7 个脚本 + launchd 调度
