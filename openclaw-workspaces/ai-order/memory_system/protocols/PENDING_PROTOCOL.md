# 未完成任务提醒机制

>每次会话结束写总结时，必须把未完成的事项写入 `projects/<项目>/problems/PENDING.md`。
> 新会话启动时，按需读取这些待办并主动提醒用户。

---

## 核心文件

```
projects/<项目>/problems/PENDING.md    ← 未完成任务追踪（核心文件）
```

---

## PENDING.md 格式

```markdown
# 待跟进事项 | YYYY-MM-DD 更新

##🔴 紧急（需立即处理）

| ID | 日期 | 待办 | 状态 | 卡点 |
|----|------|------|------|------|
| P-001 | 06-05 | BOSS Cookie 刷新 | 🟡进行中 | 待用户确认 |
| P-002 | 06-04 | PROD-A018 库存补货 | 🔴待处理 | 等待采购确认 |

## 🟡 进行中

| ID | 日期 | 待办 | 状态 |
|----|------|------|------|
| P-003 | 06-03 | iCloud 代理方案 | 🟡进行中 | TLS问题待解决 |

## ✅ 已完成（本周）

| ID | 日期 | 完成日期 | 备注 |
|----|------|----------|------|
| P-000 | 06-01 | 06-05 | Skill版本管理上线 |
```

---

## 规则

### 会话结束
1. 从本次会话的「待跟进」提取未完成的
2. 追加到 `projects/<项目>/problems/PENDING.md`
3. 已完成的从「进行中」移到「已完成」

### 新会话启动（按需）
- 提到某项目时 →读 `PENDING.md`，检查有无紧急事项
- 有紧急事项 → 主动告知用户
- 有过期很久的待办 → 询问是否继续或放弃

### 每日17:00 日结
1. 检查所有项目的 `PENDING.md`
2. 有超期项（>3天无更新）→ 飞书提醒

---

## ID 命名规则

格式：`P-<序号>`

- P-001 ~ P-099：主 session
- 各 sub-agent 用各自前缀（如 ai-order-P-001）

同项目内 ID 唯一，不得重复。序号由记录者自行维护。
