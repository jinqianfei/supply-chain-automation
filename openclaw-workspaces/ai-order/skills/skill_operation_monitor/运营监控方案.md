# Skill 运营监控方案（基于订单处理Skill分析）

> 基于 `skill_order_to_huading_template` 实际情况设计的运营监控方案

---

## 一、现状分析

### 测评报告摘要（2026-06-04）

| 指标 | 数值 | 说明 |
|------|------|------|
| 测试总数 | 65条 | 来自4个测试集 |
| 准确率 | 61.5% | 40/65 matched |
| 高置信度(≥0.8) | 98.5% | 64/65 |
| 低置信(<0.7) | 1条 | 1.5% |
| 未匹配 | 1条 | "不存在商品XYZ123"（预期） |

**核心问题**：测试集有重复商品（同一商品同时存在大单位/小单位），导致一半匹配到错误单位。

### 订单处理Skill架构

```
Step 1: parse()       → 订单解析（Excel/图片/PDF/文字）
Step 2: transform()   → 字段标准化
Step 3: _match_store() → 门店匹配（⚠️ 用户确认点）
Step 4: _match_sku()  → SKU映射（⚠️ 用户确认点）
Step 5: _generate_*   → 生成Excel
```

**关键特征**：
- 有两个用户确认节点（门店确认、SKU确认）
- 5层SKU匹配逻辑（Layer 0别名 → Layer 1精确 → Layer 2模糊 → Layer 3分词）
- 处理结果有 matched/unmatched 状态
- 匹配有置信度分数

## 二、监控指标设计

### 2.1 核心指标（与业务强相关）

| 指标 | 计算方式 | 阈值 | 用途 |
|------|---------|------|------|
| **订单处理量** | 每日/周/月计数 | — | 活跃度 |
| **门店匹配成功率** | matched_store / total | < 90% 警告 |
| **SKU匹配成功率** | matched / total items | < 85% 警告 |
| **高置信度占比** | ≥0.8 / total | < 90% 警告 |
| **低置信度占比** | <0.8 / total | > 10% 警告 |
| **用户确认率** | confirm / (auto + confirm) | > 30% 需优化 |
| **自动完成率** | auto / total | 越高越好 |

### 2.2 扩展指标（问题诊断用）

| 指标 | 计算方式 | 用途 |
|------|---------|------|
| **Layer分布** | 各层匹配次数/总匹配 | 判断匹配瓶颈在哪层 |
| **未匹配商品Top** | 未匹配商品名分组计数 | 决定优先补充哪些别名 |
| **未匹配门店Top** | 门店维度分组计数 | 发现门店数据问题 |
| **用户跳过率** | skip / total | 判断用户体验 |
| **用户纠正率** | correct / total | 发现匹配逻辑问题 |
| **处理时长** | avg(processing_time_ms) | 性能监控 |

```json
{
  "timestamp": "2026-06-04T17:30:00",
  "order_id": "ORDER_20260604_001",
  "skill_version": "5.8",

  "sku_matching": {

      "layer0_alias": 3,

  "processing_time_ms": 4521,
  "error": ""

```

### 3.2 user_action 枚举

| 值 | 含义 |
|-----|------|
| `auto` | 全自动处理完成 |
| `confirm` | 用户确认了匹配结果 |
| `skip` | 用户跳过了该项目 |
| `correct` | 用户纠正了错误匹配 |

### 3.3 关键设计决策

**为什么不记每条商品的明细？**
- 日志量太大（一个订单可能50+商品）
- 汇总统计足够判断整体质量
- 未匹配商品通过 Top 统计发现即可

**为什么分层统计 Layer分布？**
- 可以精确定位匹配瓶颈
- 如果 Layer0 alias 命中率高，说明别名表覆盖好
- 如果 Layer3 分词命中多，说明模糊匹配需优化

## 四、迭代决策规则

### 4.1 决策矩阵

| 维度 | 正常 | 警告 | 严重 | 决策 |
|------|------|------|------|------|
| SKU匹配率 | ≥ 90% | 80%~90% | < 80% | 🟢监控 / 🟡计划升级 / 🔴立即升级 |
| 低置信度占比 | < 5% | 5%~15% | ≥ 15% | 同上 |
| 用户确认率 | < 20% | 20%~40% | ≥ 40% | 🟡优化体验 |
| 未匹配商品集中度 | 分散 | Top3占80% | 同上 | 🟡补充别名 |

### 4.2 迭代优先级

```
🔴 P0 立即处理：
  - SKU匹配率 < 75%
  - 同一商品持续匹配失败 > 10次/周
  - 用户纠正率 > 15%（匹配逻辑系统性错误）

🟡 P1 计划处理：
  - SKU匹配率 75%~85%
  - 低置信度占比 > 10%
  - Top 3 未匹配商品占比 > 50%

🟢 P2 持续观察：
  - SKU匹配率 ≥ 85%
  - 各项指标稳定
  - 用户干预率 < 20%
```

## 五、实施方案

### 5.1 集成方式

**方案选择**：B）集成到建单流程（推荐）

理由：
- 不需要单独启动Skill
- 处理订单时自动记录日志
- 对金姐完全透明，不增加操作负担

**集成位置**：`skill_order_to_huading_template/__init__.py` 的 `execute()` 方法末尾

```python
def execute(self, order_input, ...):
    # ... 现有逻辑 ...
    
    # ========== 记录运营日志 ==========
    self._log_operation({
        "order_id": order_id,
        "result": result,  # 处理结果
        "meta": meta,       # 处理时长、商品数等
    })
    
    return result
```

### 5.2 数据存储

**方案选择**：A）JSONL（当前阶段足够）

理由：
- 单个建单Skill日志量不大（每天几十到几百条）
- 实现简单，无需数据库运维
- 查询用 Python 脚本够用

### 5.3 触发方式

**方案选择**：C）每日定时 + 按需查询

- 每日 17:00 生成日报（金姐下班前看一眼）
- 按需查询特定问题（用户反馈时诊断用）

## 七、与测评Skill的分工

| Skill | 职责 | 触发时机 |
|-------|------|----------|
| **skill_openclaw_test** | 上线前测试验证 | 每次代码改动后 |
| **skill_operation_monitor** | 上线后运营跟踪 | 每天/每周持续 |

```
代码改动
    ↓
skill_openclaw_test 测评
    ↓
通过后部署
    ↓
skill_operation_monitor 监控
    ↓
发现问题 → 修复 → 测评 → 部署 → 监控（循环）
```
